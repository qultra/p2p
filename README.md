외부 라이브러리 리소스에 의존하지 않고 오직 내부 코드만으로 HTML 파일 하나만으로 웹에서 가장 안전한 메신저를 개발할 수 있을까?
이 궁금증 하나에서 출발했던 실험 프로젝트는 실사용 가능한 수준의 웹사이트가 되었습니다.

지원 버전이 낮은 구형 기기에서도 최신 언어 기반 앱을 실행하기 어려운 오래된 하드웨어에서도
소프트웨어 지원이 중단되고 업데이트가 끊기고 어플리케이션 설치가 안 되는 운영체제나 플랫폼에서도

크롬 OS나 교육 전용 기기처럼 앱 설치가 제한된 환경에서도
옛날 컴퓨터처럼 아무리 제한이 많아도 인터넷만 된다면 웹사이트는 접속할 수 있죠

그저 네트워크에 연결할 수만 있다면 모두가 사용할 수 있는 방식으로
안전하게 대화할 수 있는 자유가 보장되어야 한다.

이 이념과 철학에서 시작했던 하나의 프로젝트가
오늘날의 상용화된 메신저 수준의 기능을 지원하고 가장 안전한 암호화를 보장한다면
여러분은 어떻게 생각 하시나요?

물론 암호화 부분에서 깃허브나 외부에서 라이브러리를 사용할 수 있게 호출할 수 있었지만
과거 해킹 사례들을 생각해보면 암호화 기능이 코드 자체에 내장되어 있지 않은 경우에는

언제든 외부에 있는 라이브러리 변동성에 의해서 프로젝트 전체 보안이 위험에 처할 수 있는 환경이기에
깃허브에 있는 코드나 외부에서 무언가를 가져오는건 공격 표면을 증가시키는 위협 요소라고 생각했습니다.

게다가 어떤 암호화는 웹에서 일부 제한적인 운영체제나 브라우저에서 안정적으로 지원하는 조건도 까다로워서
구형 기기나 특정 상황에 따라서 제대로 작동하지 않을 수 있는 경우도 존재했죠...

저는 이런 문제를 해결하고 싶었습니다.

모두가 말했습니다,
웹에서는 안전한 통신이 불가능하다.

IRC와 같은 프로토콜이나 인터넷 기반의 채팅은 기본적으로 암호화가 지원되지 않고
플러그인조차 제한적인 경우에만 가능하다.

이메일조차도 당연히 암호화 되지 않고 서버가 모든 내용을 볼 수 있는 웹 기반 통신이다.

S/MIME에서조차도 취약한 부분이 존재하고, PGP는 웹에서만 사용하기엔 거의 불가능에 가깝고
이를 지원하는 소프트웨어나 플러그인을 설치해서 지원되는 클라이언트에서 사용해야 한다.

모두 맞는 말이었습니다.
유일하게 웹사이트 PGP 암호화를 사용할 수 있는건 브라우저 확장 프로그램을 설치하는 방법 뿐이지만

그것조차도 모바일이나 특정 OS와 같은
웹 확장 프로그램 설치가 제한적인 환경에서는 사용이 불가능하며,

게다가 특정 브라우저나 웹사이트에서는 이러한 웹 확장 프로그램을 차단하거나
mailvelope 같은 PGP 암호화 기능을 지원하지 않는 경우도 존재하니까요.

또한 웹 확장 프로그램 설치는 브라우저 핑거프린팅에 취약해지는 문제도 있죠...

S/MIME는 일반인들이 사용하기에는 복잡한 인증서를 발급하며 이는 프라이버시에 좋지 않고

게다가 발견된 취약점은 물론이고 심지어 인증서 발급 비용이 존재하고 특정 기관을 신뢰해야 하거나
암호화를 위한 인증서를 사용할 수 있는 그 기간조차도 제한을 두는 경우가 많아요...

특히 애플은 3년 정도까지만 제한을 걸어두고 있기에 그 이상의 기간을 가진 인증서는 지원되지 않아요

저도 웹과 암호화를 좋아하지만 S/MIME의 위와 같은 사용성 문제 땜에 고민을 많이 했었죠 ㅠ

웹 자체는 기본적으로 암호화 되지 않으며 사용자가 직접 암호화를 하는 경우에도
플랫폼에 정보가 존재하며 관련 내용 또는 메타데이터 등을 서버에서 볼 수 있다.

이게 무슨 말이냐면,

계정을 특정하고 플랫폼 서버 자체에서 검열이나 감시가 가능하다는 뜻이에요.

일반적으로는 서버 하드웨어를 암호화 해도 관리자 키를 가진 경우에는 모든 내용을 서버에서 확인할 수 있고,
또한 관리자는 시스템 권한을 통해 사용자의 암호화를 무효화 시킬 수 있어요.

이것은 암호화된 이메일 제공자인 투타노타가 법원 명령에 의해서
평문으로 도착하는 모든 메시지를 서버에서 암호화하지 않고 가로채서
이메일의 내용을 암호화되지 않은 일반 텍스트로 볼 수 있게 한 것과 같죠.

웹 기반 통신 서비스는 이와 같이 실제로 상당히 위험한 부분이 많이 존재하는게 맞아요.

전체 백엔드 소스코드를 공개하는 경우도 거의 없지만 모든 서버 코드를 100% 공개하는 경우에도 마찬가지로

본인이 직접 셀프호스팅을 사용하지 않는 한 서비스가 실제로 뒤에서 어떻게 운영하고 있는지 투명하게 볼 수 없는 경우가 대부분이죠.
특정 기업이나 업체 제공자 플랫폼을 신뢰해야만 하는 이와 같은 문제점들이 존재하는 웹에서 어떻게 해야 안전한 통신이 가능할까요?

특히 웹페이지 하나만으로 완벽하게 채팅에 통신에 연락까지 가능하다?
여러분이 사용하는 대부분의 사이트는 https 암호화를 사용하죠
하지만 이건 플랫폼 서버에서 검열하고 감시할 수 있어요.

서버로 전송하는 내용을 중간에서 가로채거나 읽기 어렵게 만들 뿐이지,
도착하는 플랫폼 서버에서는 모든 내용을 다 볼 수 있어요.

즉, 내용을 서버에서도 볼 수 없게 암호화 되는게 아니에요.
게다가 웹 플랫폼을 이용하려면 대부분 계정이나 사용자 정보가 필요하죠.

저는 암호화, 프라이버시에 관심이 많았어서 보안을 강화하기 위해서 오랜 기간 동안 배우고 고민했어오
제가 처음으로 이메일에 PGP 암호화를 사용했던게 2018년이니까 초등학생 때부터 관심을 가지고 프로그래밍 언어를 배웠었죠.

저는 단순히 연락을 위한 메신저라면 사용자에 대한 정보가 없었으면 좋겠어요.
가입할 필요가 없고 그 사람에 대한 정보를 보관하지 않고

ID나 계정이 존재하지 않으면 누구인지도 모르는거죠

연락을 주고받는 사람들끼리만 서로 알면 되는거지

불필요한 정보를 보관하거나 할 필요가 전혀 없죠.


중앙에 고정된 하나의 데이터를 저장하는 서버라는 개념 자체가 존재하면 안 되고
민감한 데이터를 서버를 통해서 전달하거나 전송해서는 안 된다.

단순히 사용자를 연결해주는 것 이외에는 어떠한 연결도 불필요하다.

저는 이 원칙을 정확히 지키면서 개발했어요.


개인의 권리와 표현의 자유를 위한 신념에서 시작했기에
모든 프로잭트는 100% 앞으로도 영원히 무료로 사용할 수 있어요


사용자의 블루투스 기기 같은 통신이나 메쉬 네트워크 같은 방식도 존재하지만
저는 이게 비현실적이라고 생각했어요

바다를 건너서 지구 반대편에 있는 다른 국가의 사람과 그런 방식으로는 절대 연결할 수 없을테니까요...

제가 생각하는 최선의 방법으로 보안, 암호화, 편의성을 실제 사용 가능한 수준까지 높여봤지만
웹에서 가능한건 한계가 있다보니까 아직 부족한 부분도 많이 있다고 생각해요.

제한적인 환경의 기기나 운영체제, 브라우저에서
소프트웨어나 펌웨어 상관 없이 누구나 제한 없이 사용할 수 있어야 하니까요
